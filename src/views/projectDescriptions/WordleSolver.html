Wordle Solver

A basic python application that employs both a greedy solution and an entropic solution to solve Wordle puzzles with optimal guesses.

Project Overview

Out of curiosity after playing wordle with my coworkers, I wondered what if I could algorithmically play the game in such a way that would guarantee a better average performance compared to human players.

At it's core being a CSP (Constrained Satisfaction Problem) designing a simplified terminal version of the game then running thousands of simulations to test different solving algorithms was a fun exercise in both programming and algorithm design.

After quickly designing a basic terminal version of the game using Python to test performance, I also implemented an automated solver that uses Selenium to play the actual web version of Wordle on New York Times' website.

After playing around with a few algorithm designs, the final result was a bot that when tested on the set of 2315 possible wordle answers, was able to attain a guess count of under four 93% of the time, with an average guess count of 3.67, while the average game score as of Nov 2025 (source) sits at around 3.80

Challenges

The initial approach to this problem was to take a greedy solution that would simply pick the word that contained the most common letters among the remaining possible answers via a frequency ranking system. This worked decently, but the successrate for 6 total guesses was only 98%. If you wanted something as fast as possible, this was extremely effective as it is an O(n) solution. However, if you wanted the least number of average guesses made, a different solution would need to be implemented.

The next approach taken was to implement an entropic solution that would calculate the expected information gain of each possible guess, and then using that ranking as a way of determining the next word. This meant that even if a guess was not possible (i.e. contains/missing letters not in the answer), it could still be used if it provided high information gain. This solution was much more effective, achieving a 93% success rate for 4 or fewer guesses, and an overall 100% effective success rate within 6 guesses for the wordle answer set.

However, this solution was much more computationally expensive, as it required calculating the information gain for each possible guess against all remaining possible answers. This brought up a new implementation that ultimately drew this project to a close, where the technical fastest way to solve the finite list of wordle answers was simply to precompute a lookup table of optimal guesses for each possible state of the game, and then simply reference that table during gameplay.

This gives both an O(1) solution and also better than human level performance at the cost of being inflexible to changes in the wordle answer set or rules.
